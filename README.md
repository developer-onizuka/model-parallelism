```
       [ 入力トークン ]
              |
      ------------------
      【 Attention層 】 <--- 全体の10%以下の重み
      ------------------
              |
      << All-Reduce >> (軽い通信)
              |
      ------------------
      【   FFN層    】 <--- 全体の90%近い重み
      ------------------
              |
      << All-to-All >> (重い重みに基づく、大量のデータ移動)
              |
```

```
【GPU 1枚(100GB)あたりのメモリ使用イメージ】

 0GB          20GB          40GB          60GB          80GB         100GB
 [-----------|---------------------------------------|-------------|-----]
      ↑                 ↑                               ↑             ↑
  Attention等        FFN / Experts                    KVキャッシュ     作業
 (TPで分割)        (EPで担当分を丸ごと保持)             (動的確保)      領域

■ 内訳の解説：
1. Attention重み (約5~10GB):
   Wq, Wk, Wv, Wo など。全層分をTPで6分割するため、1枚あたりは意外と小さい。
   
2. FFN / Experts (約60~70GB): ★ここが支配的
   MoEの場合、担当する専門家（例：32個中5〜6個）の重みを丸ごと載せるため、
   メモリの大部分をここが占有します。

3. KVキャッシュ (約20~30GB):
   推論が進むにつれて増えていく領域。ここを使い切ると「長い文章」が読めなくなる。

4. 作業領域/バッファ (約5GB):
   All-ReduceやAll-to-All通信、計算中の中間値を置くための「机」。
```

# DP(2) × MP(2) × EP(4) + 分離型KVキャッシュの例
モデル120GB（Attn: 40GB / FFN: 80GB）に対し、100GBメモリを積んだ4基のGPUでのメモリ配置図

```
[ リクエスト A ]               [ リクエスト B ]
              |                             |
    +---------+---------+         +---------+---------+
    | 【GPU 0】 | 【GPU 1】 |         | 【GPU 2】 | 【GPU 3】 |
    | (100GB)  | (100GB)  |         | (100GB)  | (100GB)  |
    +---------+---------+         +---------+---------+
    | [Attn 1/2]| [Attn 1/2]|         | [Attn 1/2]| [Attn 1/2]|
    | (MP:20GB)| (MP:20GB)|         | (MP:20GB)| (MP:20GB)|
    +---------+---------+         +---------+---------+
    | [Exp 0,1]| [Exp 2,3]|         | [Exp 4,5]| [Exp 6,7]|
    | (EP:20GB)| (EP:20GB)|         | (EP:20GB)| (EP:20GB)|
    +---------+---------+         +---------+---------+
    | [KV A]   | [KV A]   |         | [KV B]   | [KV B]   |
    | (60GB)   | (60GB)   |         | (60GB)   | (60GB)   |
    +---------+---------+         +---------+---------+
          |         |                     |         |
    [ Attention (MP) ]             [ Attention (MP) ]
    ( 0,1ペアでAを計算 )             ( 2,3ペアでBを計算 )
          |         |                     |         |
    [ AのKVを読み書き ]               [ BのKVを読み書き ]
          |         |                     |         |
    +-------------------------------------------------+
    | [ Router / All-to-All ] (全Expertへトークンを配送)    |
    +-------------------------------------------------+
          |         |                     |         |
    [ FFN (EP) : 全GPUの専門家 0-7 を利用して計算 ]
          |         |                     |         |
    +---------------------------------------+
    | [ All-to-All 通信 ]                     |
    +---------------------------------------+
          |         |                     |         |
      [回答 A]  [回答 A]               [回答 B]  [回答 B]
```

特に、LLMの推論においてはステートフルな性質があるため、同じユーザー（同じ会話セッション）のリクエストを同じGPUに送るセッション維持（スティッキーセッション）は、パフォーマンス向上のための定石となります。

なぜ同じGPUに送るべきなのか、その最大の理由はKVキャッシュ（対話の記憶）にあります。

同じGPUに送るべき最大の理由：KVキャッシュの再利用
LLMとの会話では、過去のやり取りをすべて含めて次の回答を生成します。

別のGPUに飛ばされた場合：
新しいGPUには過去の会話データ（KVキャッシュ）がありません。そのため、これまでの長い会話履歴をイチからすべて計算（Prefill）し直す必要があり、応答が遅れ、計算リソースも無駄に消費します。

同じGPUに飛ばされた場合：
そのGPUのメモリ上には、前回のやり取りで作られたKVキャッシュが残っています（キャッシュアウトされていなければ）。前回の続きから計算を開始できるため、最速で回答（デコード）を開始でき、計算コストも大幅に抑えられます。


レプリカ間のロードバランス（L7負荷分散）
データ並列（DP）で並んでいる複数の「GPUセット（レプリカ）」に対し、ユーザーIDやセッションIDに基づいて、常に同じレプリカへ誘導します。

手法: 一貫性ハッシュ（Consistent Hashing）や、Cookieベースのスティッキーセッション。


1. モデル並列（TP/EP）の実装：モデルデプロイ設定
Vertex AIで自作モデルやLlama 3などのオープンモデルをデプロイする場合、deploy APIのインスタンス設定でこれらを制御します。

テンソル並列（TP）の指定: デプロイ時の引数で tensor_parallel_size を指定します。例えば、A100（80GB）を4枚使う設定にした場合、ここを 4 に設定することで、Vertex AIが内部でモデルの重みを自動的に4分割（TP）して各GPUに配置します。

エキスパート並列（EP）: MixtralなどのMoEモデルをデプロイする場合、Vertex AIがバックエンドで使用している推論エンジン（vLLMなど）が、指定されたGPU数に合わせて自動的にEPを構成します。

2. データ並列（DP）の実装：オートスケーリング
Vertex AIにおけるデータ並列は、レプリカ（Replica）という単位で管理されます。

レプリカ数: 「GPU 4枚で1つのモデルを動かす構成」を1つのレプリカと呼び、トラフィックが増えるとこのレプリカセットが丸ごと「2セット、3セット...」と横に増えていきます。これがデータ並列の正体です。

トラフィック分割:
Vertex AIの予測エンドポイント（Endpoint）がロードバランサとして機能し、空いているレプリカにリクエストを自動で振り分けます。

3. KVキャッシュとスティッキーセッションの扱い
ここが最もクラウドネイティブな部分です。

Vertex AI Prediction (GA): 標準的なエンドポイントでは、リクエストは「最も空いているレプリカ」に飛びます。そのため、厳密なスティッキーセッション（同じユーザーを常に同じ物理GPUに送る）の保証は、標準のロードバランサレベルでは設定が難しい場合があります。

vLLM / Tritonとの統合: Vertex AI上で vLLM をコンテナとして動かす構成が推奨されています。vLLMには PagedAttention という技術が組み込まれており、物理的にどのGPUに飛んでも、あるいは同一GPU内で複数のリクエストが混ざっても、KVキャッシュをメモリ内で効率的に再利用・断片化防止する仕組みが動いています。

これに加えて、ngress / Service Mesh による L7 スティッキーセッション
標準的なWebアプリと同様に、ロードバランサレベルで「同じユーザーを同じPod（GPUレプリカ）」に固定する方法とを組み合わせる。

実装方法: * GCE Ingress (GCP LB): アノテーション networking.gke.io/sticky-sessions を使用して、Cookieベースのスティッキーセッションを有効にします。

Istio / Anthos Service Mesh: DestinationRule で consistentHash を設定し、HTTPヘッダー（例：Authorization やカスタムの X-User-ID）に基づいたルーティングを行います。

「固定されたユーザー（スティッキーセッション）」×「PagedAttention」の組み合わせは、単なる「速さ」の追求ではなく、「ビジネスとしての利益率（1リクエストあたりの原価）」を劇的に改善するための必須テクニック
