# モデル並列の考え方

モデル並列とは、巨大なモデルが単一のGPUメモリに収まらない場合に、モデルを分割して複数のGPUへ分散配置する手法である。代表的な手法にはテンソル並列（TP）やパイプライン並列（PP）がある。特にテンソル並列は、計算過程でGPU間のAll-Reduce通信が発生するため通信負荷が高まりやすく、スループットを維持するためにバッチサイズを小さく抑える運用が適している。
一方で、モデルが単一GPUに収まる場合であっても、並列化の検討は不可欠である。特定のリクエストが1つのGPUに集中すると、Attention機構に伴うKVキャッシュがメモリを圧迫し、スケーラビリティの限界を招くためである。データ並列（DP）を導入し、リクエストごとにKVキャッシュを異なるGPUメモリへ分散配置するなど、推論効率と拡張性の両面から最適な並列化構成を設計することが重要である。<br>


# 1. Prefill処理
### 1-1. Attention層
一般に、入力トークンはEmbedding層でベクトル化された後、Attention層を経て文脈的な意味を付与される。Attention層では **$W_q, W_k, W_v$** の各行列が保持されており、これらとトークンとの行列積によって **$Q, K, V$** が算出される。モデル並列の一種であるテンソル並列（TP）を用いると、これら重み行列 $W$ を分割して複数のGPUへ分散配置することが可能である。これにより、計算負荷の高いPrefill処理を並列化し、効率的に $Q, K, V$ を算出できる。最終的に、各GPUで計算された結果（最終的な $Q, K, V$ ベクトルのうち、各GPUが担当する特定の列に相当する部分的な行列積の結果）はAll-Reduceアルゴリズムを用いた集団通信によって全GPU間で同期され、すべてのGPUが同一かつ完全な $Q, K, V$ データを保持した状態となり、これを用いて文脈情報を反映した埋め込み表現を各GPUが並列して（冗長に）計算する。これにより、次層への通信待機コストを最小化し、システム全体の処理効率を最適化している。<br>

<img src="https://github.com/developer-onizuka/model-parallelism/blob/main/Allreduce.gif" width="480">

### 1-2. Feed Forward Network層
Attention層を抜けて再計算された、文脈情報を反映した埋め込み表現は、Feed Forward Network（FFN）層に入力として渡される。FFN層はモデル全体の約9割（本稿の例では360GB）という膨大な重みを占めており、単一のGPUメモリには収まりきらない。そのため、ここでもテンソル並列による重みの分割が不可欠となる。具体的には、FFNを構成する2つの大きな行列（通常はアッププロジェクションとダウンプロジェクション）を分割して各GPUに配置する。入力された埋め込み表現に対し、各GPUが自身の持つ分割された重みを用いて部分的な活性化を計算し、All-to-Allで同期することで、最終的な層の出力として、各トークンの特徴をさらに深く抽象化・変換した新しい埋め込みベクトルが得られる。<br>

<img src="https://github.com/developer-onizuka/model-parallelism/blob/main/AlltoAll.png" width="520">

### 1-3. MoE
なお、このFFNは通常MoE（Mixture of Experts：混合専門家）と呼ばれる手法で構成されることが多い。MoEでは、FFN層を一つの巨大なネットワークとして扱うのではなく、多数の独立したエキスパート（専門家）と呼ばれる小規模なネットワークの集合体として定義する。各トークンがFFN層に入力される際、ルーターと呼ばれるゲート機構が、そのトークンの特徴に最も適した少数のエキスパートを選択して計算を割り振る。この仕組みにより、モデル全体のパラメータ数（360GB分）を巨大化させつつも、実際に一回の推論で動かす計算量（アクティブ・パラメータ数）を低く抑えることが可能となる。並列化の観点では、各GPUに異なるエキスパートを配置するエキスパート並列（EP）が採用される。

# 2. Decode処理
FFN層を経て特徴が強化された埋め込みベクトルは、全レイヤーの処理を終えると、モデルが持つ膨大な語彙のリストとベクトルを照らし合わせ、次にどの文字（トークン）が来るべきかの確率計算が行われる。この計算結果に基づき、最も確率の高いもの、あるいは確率分布に従って選ばれた文字が一つ出力される。出力された文字は再び入力へと戻され、次の文字を予測するための材料となる。このプロセスを繰り返すことで、最終的に一連の文章が生成される。<br>

```

              |

      ------------------
              |

              |
      ------------------
      【   FFN層    】 <--- 全体の90%近い重み
      ------------------
              |
      << All-to-All >> (重い重みに基づく、大量のデータ移動)
              |
         　Decoderへ
```

# 補足4.

### テンソル並列（Row Parallelism）における All-Reduce の仕組み

TransformerモデルのAttention機構において、出力投影行列（ $W_O$ ）に行並列（Row Parallelism）を適用した際の計算プロセスと、なぜそこで **All-Reduce** が必要なのかを具体的な数値例で解説する。

### (1) 本来計算したい式を $Y = ZW$ とする。
- **入力ベクトル $Z$**: $1 \times 2$ 行列
- **重み行列 $W$**: $2 \times 2$ 行列

$$
Z = \begin{pmatrix} 10 & 20 \end{pmatrix}
$$

$$
W = \begin{pmatrix}
1 & 2 \\ 
3 & 4 
\end{pmatrix}
$$

通常の一括計算（シングルGPU）では、結果 $Y$ は以下の通りとなる。

$$
Y = \begin{pmatrix} (10 \times 1 + 20 \times 3) & (10 \times 2 + 20 \times 4) \end{pmatrix} = \begin{pmatrix} 70 & 100 \end{pmatrix}
$$

### (2) 2枚の GPU に処理を分散させるため、重み $W$ を **行方向（横方向）** に分割し、入力 $Z$ も対応する要素ごとに分割して各 GPU に持たせる。

**GPU 0 の担当データ** <br>
- **分割された入力**: $z_1 = 10$
- **$W$ の 1 行目**:

$$
W_{row0} = \begin{pmatrix}
1 & 2
\end{pmatrix}
$$

**GPU 1 の担当データ** <br>
- **分割された入力**: $z_2 = 20$
- **$W$ の 2 行目**:

$$
W_{row1} = \begin{pmatrix}
3 & 4
\end{pmatrix}
$$

各 GPU は通信を行わずに、手元のデータだけで行列演算を行う。この時点での計算結果を **部分和（Partial Sum）** と呼ぶ。

**GPU 0 の計算結果 ($Y_0$)** <br>

$$
Y_0 = 10 \times \begin{pmatrix} 
1 & 2 
\end{pmatrix} = \begin{pmatrix} 
10 & 20 
\end{pmatrix}
$$

**GPU 1 の計算結果 ($Y_1$)** <br>

$$
Y_1 = 20 \times \begin{pmatrix} 
3 & 4 \end{pmatrix} = \begin{pmatrix} 
60 & 80 
\end{pmatrix}
$$

> [!IMPORTANT]
> この時点では、どちらの GPU も最終的な答え（70, 100）を持っておらず、データが断片化している状態。


### (3) 次のレイヤー（Layer Norm や FFN 等）の計算に進むためには、すべての GPU が「正しい合計値」を持っている必要があるため、ここでようやく **All-Reduce (SUM)** を実行することになる。

**ステップ 1: Reduce (集約演算)** <br>
全 GPU の部分和を要素ごとに足し合わせる。

$$
\begin{pmatrix} 10 & 20 \end{pmatrix} + \begin{pmatrix} 60 & 80 \end{pmatrix} = \begin{pmatrix} 70 & 100 \end{pmatrix}
$$

**ステップ 2: All (全ノード共有)** <br>
計算された合計値をすべての GPU に配布し、同期させる。

**最終ステータス** <br>
**GPU 0:**

$$
\begin{pmatrix} 
70 & 100 
\end{pmatrix} 
$$

**GPU 1:**

$$ 
\begin{pmatrix} 
70 & 100 
\end{pmatrix} 
$$

これで、分散していた計算結果が完全に統合され、全 GPU が同じ値を持って次の処理へ進めるようになる。

# 3. モデル並列　＆　データ並列
モデル並列の他にデータ並列という考え方があり、複数のリクエストを処理するためにはデータを異なるGPUグループへと分散させ、並列に処理を行うことが一般的である。具体的には、巨大な一つのモデルを全GPUで共有しつつ、複数のリクエスト（データ）を各GPUペアに割り振ることで、システム全体のスループットを向上させる。以下の図は、120GBのモデル（Attention層：40GB、FFN層：80GB）に対し、100GBメモリのGPU 4基を用いた並列化構成を示したものである。単一のGPUではモデル全体をロードできないため、2基のGPUを1ペアとしたモデル並列によってメモリ制約を克服している。同時に、複数のリクエストを並行処理するために、このペアを複数用意するデータ並列の考え方を導入した。このハイブリッド構成により、**巨大なモデルが1枚のGPUに収まらないという物理的制約と、多くのリクエストを効率的に処理したい**という実運用上の課題を同時に解決するための並列手法である。<br>

```
        [ リクエスト A ]                    [ リクエスト B ]
               |                                  |
          GPUセット#1                          GPUセット#2
   +-----------+-----------+          +-----------+-----------+
   | [ GPU 0 ] | [ GPU 1 ] |          | [ GPU 2 ] | [ GPU 3 ] |
   | (100GB)   | (100GB)   |          | (100GB)   | (100GB)   |
   +-----------+-----------+          +-----------+-----------+
   | [Attn 1/2]| [Attn 1/2]|          | [Attn 1/2]| [Attn 1/2]|
   | (MP:20GB) | (MP:20GB) |          | (MP:20GB) | (MP:20GB) |
   +-----------+-----------+          +-----------+-----------+
   | [Exp 0,1] | [Exp 2,3] |          | [Exp 4,5] | [Exp 6,7] |
   | (EP:20GB) | (EP:20GB) |          | (EP:20GB) | (EP:20GB) |
   +-----------+-----------+          +-----------+-----------+
   | [KV A]    | [KV A]    |          | [KV B]    | [KV B]    |
   | (60GB)    | (60GB)    |          | (60GB)    | (60GB)    |
   +-----------+-----------+          +-----------+-----------+
         |           |                      |           |
    [  Attention (MP)  ]                 [  Attention (MP)  ]
    (  0,1ペアでAを計算  )                 (  2,3ペアでBを計算  )
         |           |                      |           |
    [  AのKVを読み書き   ]                 [  BのKVを読み書き   ]
         |           |                      |           |
   +----------------------------------------------------------+
   | [ Router / All-to-All ] (全Expertへトークンを配送)          |
   +----------------------------------------------------------+
         |           |                      |           |
        [    FFN (EP) : 全GPUのエキスパート 0-7 を利用して計算   ]
         |           |                      |           |
   +----------------------------------------------------------+
   |                    [ All-to-All 通信 ]                    |
   +----------------------------------------------------------+
         |           |                      |           |
      [回答 A]     [回答 A]                [回答 B]     [回答 B]
```

### 3-1. スティッキーセッション
上図に示す通り、各リクエストは特定のGPUペアと結びつき、KVキャッシュを継続利用する。LLMの推論プロセスは本質的にステートフルな性質を持つため、同一ユーザー（同一セッション）のリクエストを常に同じGPUへと誘導するセッション維持（スティッキーセッション）は、パフォーマンス最適化において不可欠な要素である。仮にリクエストが別のGPUへ振り分けられた場合、そのGPUには過去の会話履歴（KVキャッシュ）が存在しない。その結果、これまでの膨大なコンテキストをすべて一から再計算（Prefill）し直す必要が生じ、応答遅延（レイテンシ）の悪化と計算リソースの浪費を招く。一方で、前回と同じGPUにルーティングされれば、メモリ上に保持された既存のKVキャッシュ（十分な領域が確保されている前提）を再利用できる。これにより、前回の続きから即座にデコード（生成）を開始でき、計算コストの大幅な削減と最速のレスポンスを実現できる。

実運用においては、データ並列で構成されたGPU群に対し、ユーザーIDやセッションIDに基づいた一貫性のあるルーティングが求められる。

- サービスメッシュ（Istio / Anthos Service Mesh）での実現: DestinationRule の consistentHash を設定し、HTTPヘッダー（例: Authorization やカスタムの X-User-ID）に基づいたルーティングを行う。

また、以下に記載するvLLMのPagedAttentionと組み合わせることで、スティッキーセッションによってユーザーを特定のGPUへ固定し、PagedAttentionによってそのユーザーの過去の会話（KVキャッシュ）をブロック単位で断片化なく保持する。これにより、メモリの空き容量を限界まで使い切りながら、再訪したユーザーに対して前回の計算結果を1ビットも無駄にせず再利用するという極限の効率化が達成される。

- PagedAttentionによるメモリ最適化: OSの仮想メモリ設計を応用し、KVキャッシュを固定サイズのブロック（ページ）として不連続な物理領域に動的配置する。これにより、メモリの断片化を最小限に抑えつつ、スティッキーセッションで戻ってきたリクエストに対しても、必要なブロックをパズルのように効率的に再割り当てすることが可能となる。


# 4. Vertex AIでの実装
### 4-1. 並列化
Vertex AIで自作モデルやLlama 3などのオープンモデルをデプロイする場合、deploy APIのインスタンス設定を通じて並列化手法を制御できる。

- テンソル並列（TP）の指定: デプロイ時の引数で tensor_parallel_size を指定する。例えば、A100（80GB）を4基使用する構成において、この値を 4 に設定すれば、Vertex AIは内部でモデルの重みを自動的に4分割し、各GPUに配置する。

- エキスパート並列（EP）: MixtralなどのMoEモデルをデプロイする場合、Vertex AIがバックエンドで使用している推論エンジン（vLLM等）が、指定されたGPU数に合わせて自動的にEPを構成する。

- データ並列（DP）: Vertex AIにおけるデータ並列は、レプリカ（Replica）という単位で自動管理される。1つのモデルを動かすGPUセットを1レプリカと定義し、トラフィックの増大に応じてレプリカセットが丸ごと増設されることで、並列処理能力を拡張する仕組みである。

### 4-2. スティッキーセッション
Vertex AI Predictionの標準的なエンドポイントでは、リクエストは最も負荷の低いレプリカへと振り分けられる。そのため、標準のロードバランサレベルで、特定のユーザーを常に同一の物理GPUへ送信する厳密なスティッキーセッションを保証することは、設定上の制約が多い。<br>

この課題に対し、Vertex AI上では vLLM をコンテナとして稼働させる構成が推奨されている。前述のPagedAttentionを活用することで、物理的にどのGPUへ割り振られた場合や、同一GPU内で複数のリクエストが混在する場合であっても、KVキャッシュをメモリ内で効率的に再利用し、断片化を防止する実装となっている。
